## 目标

- 从根本上降低同库并发写入导致的 `fast-crawler Lock wait timeout exceeded (1205)`。
- `metrics-reconcile` 在连续轮空时：休息 10 分钟后再继续。

## 现状与根因判断

- 你目前 MySQL 为 `8.0+`，并发规模中等（`fast-crawler`/`metrics-reconcile` 可能多实例）。
- `1205` 通常表示：一个事务在等待另一个事务释放行锁/间隙锁，等待超过 `innodb_lock_wait_timeout`。
- 本工程里竞争写主要来自：
  - `fast-crawler` upsert `articles` / `author_sources`。
  - `metrics-reconcile` 更新同一批 `articles` 的指标、以及 `author_sources.followers/last_crawled_at`。
- 关键点：metrics-reconcile/fast-crawler 任何一方在一次轮次里“提交过晚、更新范围过大”，都会放大锁冲突窗口。

## 改造方案（不依赖猜测的可验证步骤）

### 1. `fast-crawler`：只在真正写入后提交，避免无意义长事务/高提交成本

文件：[app/fast_crawler.py](app/fast_crawler.py)

- 在 `_filter_and_upsert()` 的 `for feed_item in feed_items:` 内增加 `did_write` 标记：
  - 只有当确实执行了 `AuthorSource`/`Article` 的新增或字段变更（包含 flush），才在 `finally/边界` 执行 `_commit_with_retry()`。
  - 跳过（continue 的那些过滤分支）不提交，减少 DB 压力。
- 同时将 `begin_nested()` 维持原意（处理唯一键冲突），但确保外层不会在整轮结束才 commit。
- 验收标准：`fast-crawler` 的 `upsert failed ... lock wait timeout` 明显减少或消失。

### 2. `metrics-reconcile`：拆分写入提交块，缩短锁持有时间窗口

文件：[tools/reconcile_article_metrics.py](tools/reconcile_article_metrics.py)

- 当前逻辑：更新完作者粉丝 + 更新完一批文章指标后，最后统一 `db.session.commit()`。
- 改造为“分段 commit”：
  - claim 阶段仍保持不变（已经有 sentinel + with_for_update/兜底 claim）。
  - 更新 `AuthorSource.followers/last_crawled_at` 完成后，**立即 commit**。
  - 再更新文章指标时：按 `commit_every_n`（例如 5 或 10）分块 commit。
- 注意：网络请求（`fetch_info_api`）与写操作的事务边界要清晰，避免把大量 UPDATE 延后到最后一次 commit。
- 验收标准：并发运行多个 worker 时，`fast-crawler` 不再频繁等待锁；metrics 的成功率/吞吐提升。

### 3. `metrics-reconcile`：连续轮空退避 10 分钟

文件：[tools/reconcile_article_metrics.py](tools/reconcile_article_metrics.py)

- 在 `main()` 的 while 循环里维护 `empty_rounds` 计数：
  - `mode=pending`：当 `reconcile_once()` 返回 `picked == 0` 时计数+1，否则清零。
  - `mode=checked-refresh`：当返回 `checked_refreshed == 0` 时计数+1，否则清零。
- 当 `empty_rounds >= 2`（可调阈值）时：`time.sleep(600)`，然后继续轮询。
- 退避期间仍保持 `--loop` 不退出。
- 可配置：新增 env 或命令行参数（例如 `--empty-threshold`、`--empty-backoff-seconds`），默认按需求写死 10 分钟。

### 4. 保持“严格不重复”的 claim 机制

- 继续保留 pending/checked 两个 claim：
  - `metrics-error` sentinel + `SKIP LOCKED`（MySQL 8 支持）+ 条件 UPDATE 兜底。
- 验收标准：多 worker 并行时同一条文章不会被重复 claim 到。

## 验证与回滚

- 验证步骤：
  1. 在测试/预发环境先开 2~~3 个 `metrics-reconcile` 实例 + 1~~2 个 `fast-crawler` 实例。
  2. 观察 `fast-crawler` 是否仍出现 `Lock wait timeout exceeded (1205)`。
  3. 检查 `metrics-reconcile` 是否在轮空时正确 10 分钟退避（日志可见）。
- 回滚：仅 revert改动（commit策略/退避）即可恢复原行为；claim 机制尽量不动。

## 预期收益

- 根本降低 InnoDB 锁等待窗口，减少 1205。
- metrics-reconcile 空转退避，减少无意义的 DB 查询与 API 请求。
